## TASK-FR — Tradução automática do francês via DeepL

Script: `scripts/translate-locale.mjs`
Config: `scripts/translate-locale.config.json`

### O que o script faz

1. Lê `src/locales/en.json` (source) + `src/locales/fr.json` (target).
2. Percorre recursivamente — para cada folha onde `fr === en` (string ainda
   por traduzir), recolhe-a para envio.
3. Embrulha termos protegidos (CPC, EMPIS, CIBEA, Algarve, …) em `<x>...</x>`.
   DeepL ignora estas tags via `ignore_tags=x`.
4. Envia lotes (default 25 strings/lote) à API do DeepL.
5. Remove as tags `<x>` após resposta e atualiza `fr.json` em memória.
6. **Backup automático** de `fr.json` para `fr.json.bak.<timestamp>` antes
   de escrever (desativar com `--no-backup`).
7. Escreve o novo `fr.json`.

### Pré-requisitos

- **Node 18+** (`fetch` nativo). Sem dependências NPM extra — só `node` e
  `fs`/`path` (já no projeto).
- **Chave DeepL Free API**:
  - Criar conta em https://www.deepl.com/pro-api?cta=header-pro-api
  - Plano "DeepL API Free" → **500.000 caracteres/mês** grátis (chega à
    larga: estimativa do FR completo é ~31k chars conforme o dry-run).
  - Chave aparece com sufixo `:fx` (free tier). Exemplo:
    `a1b2c3d4-e5f6-7890-abcd-ef1234567890:fx`
  - Se for chave Pro (sem `:fx`), o script deteta e usa o endpoint Pro.

### Como rodar

```bash
# 1. Confirmar o plano (não chama API, não escreve):
node scripts/translate-locale.mjs --dry-run

# 2. Definir a chave e correr a tradução real:
export DEEPL_API_KEY="xxxx-xxxx-xxxx:fx"     # Linux/macOS
$env:DEEPL_API_KEY = "xxxx-xxxx-xxxx:fx"     # PowerShell
node scripts/translate-locale.mjs
```

### Opções do CLI

| Flag | Default | Descrição |
|---|---|---|
| `--dry-run` | — | Não chama API, não escreve. Só imprime plano por secção. |
| `--source <lang>` | `en` | Locale fonte (lê `src/locales/{lang}.json`). |
| `--target <lang>` | `fr` | Locale alvo. |
| `--no-backup` | — | Não cria `.bak.<timestamp>` antes de gravar. **Não recomendado**. |
| `--help`, `-h` | — | Mostra esta ajuda. |

### Estimativa de custo (Free API)

| Caracteres totais | **~31.000** |

## Termos protegidos

`scripts/translate-locale.config.json` define `protectedTerms` — palavras
e siglas que **não** devem ser traduzidas (nomes próprios, marcas, locais).

Lista atual (2026-05-26):

```
CPC, Connecting People & Companies, EMPIS, CIBEA, Algarve, Lisboa, Lisbon,
Faro, Albufeira, Portugal, Português, Portuguese, Portugais, Portuguesa,
Neopulse, Firebase, Firestore, Google, WhatsApp, Carla, Sarah, Mateo,
Silva, Vol1, OP2, OP3, T1, SCAS, PDI, NIF, CEP, SNS
```

Para adicionar novos termos: editar o array `protectedTerms` no JSON.

### Como o script protege

Cada termo é embrulhado em `<x>termo</x>` antes do envio. O DeepL recebe
`tag_handling=xml` + `ignore_tags=x` e devolve o conteúdo dentro de `<x>`
inalterado. As tags são removidas após a tradução.

Exemplo:

```
Antes (en):    "Welcome to CPC in Algarve, Portugal."
Embrulhado:    "Welcome to <x>CPC</x> in <x>Algarve</x>, <x>Portugal</x>."
DeepL devolve: "Bienvenue à <x>CPC</x> en <x>Algarve</x>, <x>Portugal</x>."
Sem tags:      "Bienvenue à CPC en Algarve, Portugal."
```

### `skipKeyPaths`

Paths específicas que nunca devem ser traduzidas (mesmo se estiverem em
EN). Atualmente: `common.languages.{pt,en,es,fr}` — porque cada idioma
mostra os outros 4 com os nomes próprios deles (no FR, "English" fica
"Anglais" — incorreto; queremos sempre o nome no idioma do próprio
botão).

## Pós-tradução: revisão obrigatória

A tradução automática é uma **base inicial**, não um produto final. Após
correr o script:

1. **Validar build:**
   ```bash
   npm run test:run -- src/lib/i18n.integrity.test.ts
   npm run build
   ```

2. **Smoke test visual:** abrir a app em `?lang=fr` ou via seletor de
   idioma e percorrer:
   - Landing pages (`/`, `/sobre`, `/como-funciona`, `/contactos`)
   - Auth (`/entrar`, `/registar`)
   - Dashboard migrante (`/dashboard/migrante`)
   - Dashboard CPC (`/dashboard/cpc`)
   - Dashboard empresa (`/dashboard/empresa`)

3. **Procurar problemas comuns** que a tradução automática produz:
   - Falsos amigos (palavras parecidas com significado diferente)
   - Sintaxe que muda significado (ordens de palavras importantes)
   - Termos técnicos do CPC mal traduzidos (ver `protectedTerms`)
   - Plural/singular inconsistente com a interpolação `{count}`
   - Tom (formal vs informal — DeepL tende ao formal "vous" em FR; CPC
     decide o tom)

4. **Backup de segurança:** o script cria `fr.json.bak.<timestamp>`
   automaticamente. Para reverter: `cp src/locales/fr.json.bak.NNN src/locales/fr.json`.

## TASK-CR — Crioulo (Kriolu)

**Estado:** estrutura criada (2026-05-26). Conteúdo aguarda CIBEA.

### Variante assumida

Cabo-verdiano (ISO `kea`) — escolhido pelo perfil migratório dominante
no Algarve. **A confirmar com CIBEA via D11** (`docs/CLIENT_DECISIONS.md`).

### Por que não há tradução automática

- **DeepL** não suporta nenhum crioulo lusófono.
- **Google Translate** suporta apenas crioulo haitiano (`ht`), que é
  uma língua diferente (base lexical francesa) — não serve para
  cabo-verdiano ou guineense.

`scripts/translate-locale.mjs` (TASK-FR) **não** deve ser usado para
`kea.json` — DeepL devolverá texto inalterado ou errado.

### Como aplicar tradução quando chegar

1. Editar `src/locales/kea.json` mantendo estrutura paralela (chaves
   alinhadas com `pt.json`). Ferramentas: editor JSON com schema lock
   (VS Code com extensão `JSON Tools`) reduz risco de drift.
2. Validar:
   ```bash
   npm run test:run -- src/lib/i18n.integrity.test.ts
   npm run build
   ```
3. Smoke test visual: `npm run dev`, abrir app, escolher "Kriolu" no
   Header, percorrer Landing + Auth + Dashboard.
4. Atualizar `src/locales/kea.json.STATUS.md` para "tradução completa".

### Se CIBEA pedir variante diferente

Crear novo locale ao lado (ex.: `pov.json` para guineense) com o mesmo
processo. Não substituir o `kea.json` — múltiplas variantes podem
coexistir, cada uma com a sua entrada em `Language` type e seletor.
